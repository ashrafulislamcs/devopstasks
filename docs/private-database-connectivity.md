# Private Database Connectivity (Task 4)

The database (Amazon RDS PostgreSQL, provisioned in `terraform/modules/rds`)
is never publicly reachable. This document explains the design.

## How EKS connects privately to the database
- Both the EKS worker nodes and the RDS instance live inside the **same VPC**
  and are placed in the **private subnets** (`terraform/modules/vpc`).
- Traffic from a backend pod to the database never leaves the VPC — it's
  routed entirely over the VPC's internal network, using the private route
  table (no NAT, no internet gateway involved for this path).
- The backend pod connects using a private hostname (see DNS below), not a
  public RDS endpoint, on port 5432.

## Private subnet / private endpoint design
- 2 **public** subnets: only used for the internet gateway route, the NAT
  gateway, and the internet-facing load balancer created by the Ingress.
- 2 **private** subnets (across 2 AZs for availability): EKS worker nodes
  and the RDS instance both live here. Private subnets have a route to the
  internet only via the NAT gateway (for pulling images/updates) — nothing
  from the internet can initiate a connection *into* these subnets.
- `aws_db_instance.this.publicly_accessible = false` is the explicit control
  that prevents RDS from ever getting a public IP, in addition to the subnet
  placement.

## Private DNS requirement
- A Route 53 **private hosted zone** (`internal.<cluster>.local`) is
  associated with the VPC, so DNS resolution for the database hostname only
  works from inside that VPC — it does not resolve on the public internet.
- A `CNAME` record (`db.internal.<cluster>.local`) points at the RDS
  instance's actual endpoint, so application config references a stable
  internal name instead of the AWS-generated endpoint, which can change on
  certain maintenance events.
- This hostname is what's injected into the backend via the `DB_HOST` key in
  `k8s/backend-configmap.yaml`.

## NSG / firewall / security group rules
- The RDS security group (`modules/rds/main.tf`) has exactly **one** ingress
  rule: TCP/5432 from the EKS node security group. No CIDR-based rule, no
  `0.0.0.0/0` anywhere.
- This means even another EC2 instance in the same VPC, if it isn't in that
  specific security group, cannot reach the database.

## How only the backend can access the database
- Security-group-to-security-group rules (above) mean only pods running on
  EKS nodes tagged with the node security group can reach port 5432 at the
  network layer.
- At the Kubernetes layer, only the `backend` Deployment is given the
  `DB_HOST`/`DB_USER`/`DB_PASSWORD` values (via ConfigMap + Secret) — the
  `frontend` Deployment has no database credentials or network path to the
  database at all, and never needs one (it only talks to the backend over
  HTTP on port 8080).
- Recommended hardening (see `docs/future-improvements.md`): a Kubernetes
  `NetworkPolicy` that further restricts DB-port egress to only pods labeled
  `app: backend`, so even a compromised non-backend pod on the same nodes
  can't reach the database at the pod-network level.

## How database credentials are stored securely
- The master password is generated by Terraform (`random_password`) and
  written directly to **AWS Secrets Manager** — never to a `.tf`/`.tfvars`
  file or console output.
- The Kubernetes Secret (`k8s/backend-secret-example.yaml`) is an *example*
  only; in practice it would be populated at deploy time from Secrets
  Manager (via the External Secrets Operator or the AWS Secrets & Configuration
  Provider for the Secrets Store CSI Driver), not hand-written or committed.
- The backend reads the password only via `secretKeyRef` at runtime — it's
  never baked into the Docker image or logged.

## How to confirm the database is not publicly accessible
- `aws rds describe-db-instances --db-instance-identifier <id> --query
  "DBInstances[0].PubliclyAccessible"` should return `false`.
- Attempt to resolve/connect to the RDS endpoint from outside the VPC (e.g.
  from a machine off the corporate VPN) — DNS resolution and/or the TCP
  connection should fail.
- Check the DB's security group in the AWS console/CLI and confirm there is
  no `0.0.0.0/0` ingress rule.
- Confirm the DB subnet group only contains private subnet IDs
  (`aws rds describe-db-subnet-groups`).
